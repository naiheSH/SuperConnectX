# 连接多线程改造

## 背景

改造前，所有连接（Telnet、COM）的数据处理都在 Electron 主线程中串行执行。当多个连接同时活跃时，一个连接的 Buffer 处理、HEX 转换、行分割等 CPU 密集操作会阻塞其他连接的数据处理，导致延迟增加和吞吐下降。

## 改造目标

- 将每个连接的协议处理（Buffer 处理、HEX 转换、行分割）移到独立线程
- 连接之间完全隔离，一个连接崩溃不影响其他连接
- 前端零改动

## 最终架构

```
┌─────────────────────────────────────────────────────┐
│  渲染进程（单线程）                                   │
│  Tab A (Telnet)   Tab B (COM)   Tab C (Telnet)      │
│       │                │              │              │
│       │           IPC (不变)          │              │
└───────┼────────────────┼──────────────┼──────────────┘
        │                │              │
┌───────┼────────────────┼──────────────┼──────────────┐
│       ▼                ▼              ▼              │
│  主线程 IpcConnector                                │
│       │                │              │              │
│  shouldUseWorker?      │              │              │
│   ✅ Telnet          ❌ COM         ✅ Telnet       │
│       │                │              │              │
│       ▼                ▼              ▼              │
│  WorkerPool       主线程直连       WorkerPool        │
│       │           ComClient            │             │
│       ▼                │              ▼              │
│  Worker A          串口硬件        Worker B          │
│  TelnetClient                     TelnetClient       │
│  (独立线程)                       (独立线程)          │
└─────────────────────────────────────────────────────┘
```

### 协议路由策略

| 协议类型 | 运行位置 | 原因 |
|---------|---------|------|
| **Telnet** | Worker 线程（1 Connection = 1 Worker） | 纯 JS 协议，完全支持 worker_threads，实现线程隔离 |
| **COM (serialport)** | 主线程 | `serialport` 是 C++ native addon，内部使用了主线程相关的数据结构，与 `worker_threads` 不兼容。在 Worker 中接收数据会导致 `ACCESS_VIOLATION (0xC0000005)` 内存访问违规，整个进程崩溃 |

## 新增/修改文件

### 新增文件

#### `src/main/pool/WorkerPool.ts`
Worker 线程池管理器。职责：
- 创建和销毁 Worker 线程（按需创建，连接断开即回收）
- `sessionId -> WorkerEntry` 映射，每个连接一个独立 Worker
- 路由主线程和 Worker 之间的消息（postMessage）
- 超时处理（30s 默认超时）
- `shutdown()` 统一清理所有 Worker

#### `src/main/workers/ConnectionWorker.ts`
Worker 线程入口脚本。职责：
- 接收主线程消息（start/send/stop/update-config/shutdown）
- 创建并管理协议客户端实例（TelnetClient/ComClient）
- 在 Worker 内完成 Buffer 处理、HEX 转换
- 通过 `parentPort.postMessage()` 将处理后的数据回传主线程

### 修改文件

#### `src/main/protocol/BaseClient.ts`
- 新增 `ILogger` 接口，支持 logger 注入
- 构造函数接受可选的 `ILogger` 参数
- `getDefaultLogger()` 改为 `require` 延迟加载 + try/catch 回退到 `console`，避免 Worker 打包时引入 `electron` 依赖

#### `src/main/protocol/ComClient.ts`
- 移除顶层 `import logger from '../ipc/IpcAppLogger'`
- 所有 `logger.xxx` → `this.logger.xxx`
- 构造函数接受可选 logger，传递给 BaseClient

#### `src/main/protocol/TelnetClient.ts`
- 同上（移除顶层 logger import，改为实例属性）

#### `src/main/ipc/IpcConnector.ts`
- 新增 `shouldUseWorker()` 判断函数：COM 连接不走 Worker，Telnet 走 Worker
- `start-connect`、`send-data`、`stop-connect`、`update-connect` 四个 handler 统一通过此函数路由
- 保留直连模式（`startConnectDirect` 等）作为 COM 和回退方案的执行路径

#### `src/main/utils/ProtocolLogger.ts`
- 日志写入方法改为异步，避免 Worker 模式下阻塞消息传递

## Worker 生命周期

```
startConnection(sessionId)
  → createWorker(sessionId)     // 创建独立 Worker
  → Worker ready（postMessage { type: 'ready' }）
  → sendToWorker({ type: 'start' })  // 启动连接
  → 数据/日志通过 worker.on('message') 回调到主线程
  → onClose → terminateWorker()  // 连接断开，立即回收
  → shutdown()                   // 应用退出，统一清理
```

## 前端兼容性

**前端零改动。** sessionId 路由、IPC 通道名称、数据格式完全不变：
- `start-connect` → 返回 `{ success, message, connId }`
- `on-recv-data` → 接收 `{ connId, data, timestamp, isHex }`
- `send-data` → 返回 `{ success, message }`
- `on-connect-close` → 接收 `connId`

## 编译注意事项

1. Worker 入口文件（`ConnectionWorker.ts`）与主进程入口（`index.ts`）共享 Rollup 构建，通过 chunk 拆分避免重复打包
2. `electron` 模块在 Worker chunk 中必须被排除（`externalizeDepsPlugin` 自动处理）
3. `BaseClient` 的 logger 延迟加载确保 Worker 打包不引入 `IpcAppLogger` 及其 Electron 依赖

## 已知限制

1. **COM 连接仍在主线程**：由于 `serialport` native addon 与 `worker_threads` 不兼容，COM 连接的协议处理仍在主线程。如有多个 COM 连接同时活跃，它们之间仍可能互相影响
2. **Worker 中的 console logger**：Worker 中的日志通过 `console` 输出，不会写入 Electron 的日志文件（日志数据通过 postMessage 回传主线程后由 ProtocolLogger 统一写入）
